# Механизм синхронизации данных при восстановлении связи (TO-BE)

## Во время офлайна

Local Edge Node продолжает обслуживать продажи автономно: Local POS Logic берёт каталог и цены из Offline Cache, проводит чек и через HAL печатает фискальный документ на ККТ и инициирует оплату на терминале. Каждая завершённая транзакция не пропадает, а помещается в локальную очередь непроведённых чеков внутри Offline Cache. Планшет при этом остаётся тонким клиентом — он работает с локальным агентом по локальной сети, наличие интернета ему не требуется.

## При восстановлении связи

Sync Component обнаруживает канал и запускает двунаправленный обмен с облачным Fiscal / Sync API. Снизу вверх он выгружает накопленную очередь офлайн-чеков, которые Fiscal / Sync API проводит в Managed DB. Сверху вниз агент получает обновления каталога, цен и правил лояльности и обновляет Offline Cache — так локальный кеш приводится к состоянию облака до следующего разрыва. Обмен идёт асинхронно, чтобы возобновление торговли в онлайне не блокировалось разбором очереди.

## Защита от дублей и потерь

Каждый офлайн-чек выгружается с устойчивым идентификатором, а приём на стороне облака идемпотентен: повторная отправка того же чека (например, если связь оборвалась в момент подтверждения) не создаёт вторую запись. Пока подтверждение из облака не получено, чек остаётся в локальной очереди — это гарантирует, что ни одна офлайн-продажа не потеряется.

Приём идемпотентной переотправки с сохранением идентификатора (Idempotency Key) описан в курсе как расширение Use Case: при восстановлении сети система переотправляет операцию с тем же ID, а получатель распознаёт повтор и не выполняет действие дважды (источник: Урок_3_3_3.md, раздел «Use Cases»). Общий паттерн размещения локального кеша и sync-компонента на объекте рядом с HAL взят из источника: Урок_3_3_1.md, раздел «Контейнерная диаграмма».
